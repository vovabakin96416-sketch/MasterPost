# Шаг 12c — ВЫВОД Content Intelligence (отчёт + экран «📈 Рост»)

Третий подшаг эпика 12 («AI-директор канала»). Ядро 12a считает, 12b/12b-2 наполняют
данными — **12c впервые показывает выводы владельцу**. Форматирование текста — здесь;
AI-нарратив голосом канала — 12d. **0 токенов, миграций быть не должно.**

## Прочитать перед стартом
- `CLAUDE.md`, `STATE.md` (12b-2 готов).
- Записи 12a / 12b / 12b-2 в `docs/ARCHIVE-PROGRESS.md` (через Grep по «Шаг 12a» и т.д.).
- Готовое ядро: `src/core/analytics/{insights,bestTime,trend,outliers,engagement,dimensions,topHours}.ts`.
- Готовый сервис: `src/services/analytics/contentIntelligenceService.ts`
  (`buildChannelIntelligence` уже отдаёт `Insights` + `latestSnapshot`).

## Что уже готово (фундамент 12c)
- `buildChannelIntelligence(prisma, channelId, tz, now?)` → `{ insights, latestSnapshot }`
  (БД-only, 0 токенов). `insights`: count, best/worst по ERR, `bestTimes` (день×слот),
  `trend` (Δ% просмотров/ERR неделя-к-неделе), `outliers`. `latestSnapshot`: subscribers,
  postCount7d, avgViews7d, avgErr7d, `topHours` (нативные лучшие часы Telegram, best-first).

## Объём

### 1. ЯДРО — эвристический советник (`src/core/analytics/advisor.ts`, чистый, тесты)
`buildAdvice(insights, snapshot)` → `Advice[]` (структурные факты-рекомендации, без текста
Telegram). Правила на готовых `Insights` + `StatSnapshotRow`:
- лучший/худший слот по ERR («Пт вечер заходит лучше всего»);
- совпадают ли свои ERR-слоты с нативными часами Telegram (совпало → уверенный сигнал,
  разошлось → пометка «мало своих данных, доверяй нативным»);
- тренд охвата (up/down/flat из `compareTrend` + Δ подписчиков между снимками, если есть);
- контентные измерения: медиа/кнопки/длина против ERR — нужен ядро-хелпер
  `contentDimensionStats` (средний ERR по `hasMedia`/`hasButtons`/бакетам `charLen`),
  вынести в `dimensions.ts` или рядом;
- выбросы — отдельной пометкой (виральный/рекламный, не показатель контента).
Только факты + приоритет. Порог «достаточно данных» (не советовать на 2 постах).

### 2. ЯДРО — форматтер отчёта (`src/core/analytics/insightsReport.ts`, чистый, тесты)
`buildInsightsReport(insights, snapshot, advice, tz)` → Markdown-строки (как
`buildWeeklyReport`): «что зашло / что провалилось / лучшее время (свои слоты + нативные
часы Telegram) / тренд охвата / рекомендации». Пустые данные → понятная заглушка.

### 3. СЕРВИС (`services/analytics/contentIntelligenceService.ts`)
`buildGrowthReport(prisma, channelId, tz)` — поверх `buildChannelIntelligence` →
`buildAdvice` → `buildInsightsReport`. БД-only, без MTProto.

### 4. ВЫВОД в двух местах (решение из плана — «И в отчёт, И в экран»)
- **Еженедельный отчёт** (`weeklyReportService.ts`): дописать секцию Insights после сырых
  чисел 7c.
- **Экран «📈 Рост»** в админ-меню: широкая кнопка в главном меню + хендлер, зовущий
  `buildGrowthReport`. Паттерн — как экран «📅 Календарь» (11a) / «📊 Аналитика».

## Не делать в 12c
- AI-нарратив голосом канала — это **12d**.
- Автоматическую перестановку расписания на оптимальное время — **отдельный будущий шаг**
  (советник сначала предлагает, владелец подтверждает; полный автопилот — позже, с
  ограждениями, т.к. один выброс не должен двигать весь план).
- Telemetr / рыночные данные — **12e** (за адаптером `MarketDataProvider`).

## Контекст: авто-подбор времени (частый вопрос)
Фундамент «узнать выгодное время под канал» готов (12a+12b+12b-2): бот считает лучшее
время двумя источниками — свой ERR по слотам (`rankPostingTimes`) + нативные часы Telegram
(`topHours`), per-channel, niche-agnostic, 0 токенов, обновляется джобом раз в сутки.
**Но сейчас это только данные в БД.** 12c их ПОКАЗЫВАЕТ (советник, не автопилот).
Автопостинг по-прежнему берёт время из `Post.time` — автоматическая перестановка постов на
оптимальный час НЕ входит в 12c и делается отдельным шагом после обкатки советника.
