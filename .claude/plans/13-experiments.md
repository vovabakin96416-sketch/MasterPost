# План 13 — Optimization + Experiment Engine (модули 6 и 8 из План.txt)

## Зачем
Шаг 12 научил систему СЧИТАТЬ («что зашло / когда публиковать / растёт ли охват»).
Шаг 13 замыкает петлю: система сознательно ПРОБУЕТ варианты (Experiment Engine),
выбирает победителя и МЕНЯЕТ стратегию следующих постов (Optimization Engine).
Принцип из План.txt: «каждый следующий пост лучше предыдущего».

## Честное ограничение Telegram
Настоящего A/B в канале НЕТ (нельзя показать половине подписчиков вариант А).
Единственный честный метод — ПОСЛЕДОВАТЕЛЬНЫЙ эксперимент: варианты чередуются
между постами (ротация), сравнивается средний ERR вариантов. Следствия:
- одно измерение за раз (при 1–2 постах/день иначе не различить);
- эксперимент занимает 2–4 недели (минимум ~5 чистых постов на вариант);
- фильтр выбросов 12a/12f переиспользуется напрямую (не учиться на мусоре).

## Решения владельца (сессия 2026-07-12)
1. Эксперименты — ТОЛЬКО над AI-постами (вариант = директива промпта). Пулы не трогаем.
2. Измерения на старте — фиксированный каталог: стиль CTA · фото есть/нет ·
   длина · стиль заголовка. По 2 варианта на измерение (минимальные выборки).
3. Применение победителя — кнопкой владельца; тумблер «авто-применять» дефолт ВЫКЛ.
4. Порядок: 13 начинаем сразу (12g остаётся в очереди, независим).

## Защита от нейрослопа (ОБЯЗАТЕЛЬНАЯ часть, не опция)
Риск: петля «оптимизируй ERR» без якорей = кликбейт → схлопывание в один стиль →
AI учится у AI. Контрмеры, зашитые в подшаги:
- **Guard-метрика (13a):** победитель по ERR при падающих подписчиках за период
  эксперимента → вердикт `suspicious`, НЕ применять (ряд подписчиков уже есть, 12e-2).
- **Квота исследования (13e):** выученная стратегия применяется не к 100% постов
  (константа, ~75%); остальные — вне неё. Против схлопывания + свежие данные.
- **Срок годности победителя (13e):** выученное решение живёт N недель, потом
  предлагается перепроверка.
- **Якорь голоса (13b):** пометка источника поста `human | ai`; примеры для тона
  канала (`getSamplePosts`) в приоритете берут человеческие посты — AI не учится
  у вчерашнего себя.

## Подшаги (1 сессия = 1 подшаг)

### 13a — ядро экспериментов (чистая математика, как 12a) ✅ эта сессия
`src/core/experiments/` (ни Telegram, ни БД, ни AI):
- `experiment.ts` — типы + каталог измерений `EXPERIMENT_DIMENSIONS`
  (4 измерения × 2 варианта, ключи + русские подписи; niche-agnostic).
- `assignVariant.ts` — детерминированная ротация: вариант по счётчику уже
  назначенных постов (чередование во времени = честность выборок).
- `evaluateExperiment.ts` — вердикт по выборкам вариантов: пул просмотров всех
  вариантов → `flagViewOutliers` (реюз 12a) → чистые посты → средний ERR на
  вариант → `continue` (мало данных) | `no_difference` (Δ% ниже порога) |
  `winner` | `suspicious` (guard-метрика: победитель при `subscriberDelta < 0`).
  Константы: `MIN_POSTS_PER_VARIANT=5`, `WINNER_THRESHOLD_PCT=15`.
Тесты `tests/experiments.test.ts` (~12).

### 13b — хранение и привязка
Миграция: таблица `Experiment` (канал, dimension, статус, даты) +
`PendingPost`/`PostMetric` += `variantKey?` + `Post`/`PostMetric` += источник
`origin` (`human|ai` — якорь голоса). Репозиторий. Назначение варианта при
постановке AI-поста в очередь. Видимой фичи нет.

### 13c — вживление в генерацию
`buildPostPrompt` += опц. `variantDirective` (текст-директива варианта из
каталога 13a). AI-пост (10b) и AI-подхват (10c) при активном эксперименте
получают вариант через `assignVariant`. `getSamplePosts` — приоритет `human`.

### 13d — экран «🧪 Эксперименты» + отчёт
Меню: запустить (выбор измерения из каталога) / остановить; прогресс
(«вариант А: 4 поста, ERR 3.1% · Б: 3 поста, ERR 4.0% · нужно ещё ~5»).
Секция в еженедельном отчёте ПН: активные эксперименты + вердикты.
Плейн-текст без Markdown-эмфазы (правило 12c).

### 13e — петля самооптимизации (Optimization Engine)
«Выученная стратегия» канала в `Setting` (без миграции): победивший вариант
на измерение + срок годности. Кнопка «применить победителя» в отчёте/экране;
тумблер авто-применения (дефолт ВЫКЛ). Применённая стратегия подмешивается в
`buildPostPrompt`; квота исследования (~75/25).

### 13f (опц., можно в бэклог) — AI-советник экспериментов
Haiku по инсайтам 12c предлагает «что тестировать следующим», владелец
подтверждает кнопкой. Общий бюджет `ai_daily_cap`.

## SaaS-нюансы (зафиксировано, работа — в Шаге 14+, см. BACKLOG)
- Выученная стратегия и якорь голоса — строго пер-канальные (разные аудитории →
  разные победители → каналы на одном движке РАСХОДЯТСЯ, а не совпадают).
- Сид-контент канала №1 НЕ тиражируется новым клиентам; общий контент — только
  шаблоны, переписываемые в голос канала при онбординге.
- Онбординг большого канала с готовой историей — ЛУЧШИЙ случай: богатая история
  постов = богатый профиль голоса (`getSamplePosts` из его же постов).
