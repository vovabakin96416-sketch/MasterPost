# Эпик 14 — мультитенант и триал («AI-директор» для чужих владельцев)

> Восстановлен 2026-07-20: оригинал, записанный в сессии 14a, не сохранился на диске
> (вероятно, сбой синхронизации OneDrive). Содержание собрано из STATE.md, архива и
> решений владельца.

## Зачем

Владелец хочет дать ОДНОМУ доверенному тестеру бесплатный доступ на его собственном
канале. Разбор показал: полный Шаг 14 роадмапа (веб-дашборд + биллинг + API) для этого
не нужен — касса бессмысленна без платящих, сайт избыточен (продукт целиком живёт в
Telegram). Реальный блокер — мультитенант (хвост Шага 9): до эпика бот был одно-арендный.

## Решения владельца

- Полный Шаг 14 делаем как эпик, но в порядке «сначала триал, потом касса».
- Триал — ТОЛЬКО через Telegram, БЕЗ сайта (веб-кабинет 14c–14d отложен).
- Тестер пользуется существующим общим ботом (свой bot-token на клиента = 14b-bis, не сейчас).
- Ритм: 1 сессия = 1 подшаг.

## Подшаги

- **14a — фундамент** ✅ (коммит `20f0919`): модель `Owner` (`telegramUserId @unique`,
  опц. `name`) + `Channel.ownerId String?` (`onDelete: SetNull`) + индекс; репозиторий
  `ownerRepository` (`ensureOwner` / `findOwnerByTelegramId` / `claimOrphanChannels`);
  на старте `index.ts`: `ensureOwner(ADMIN_ID)` → бэкофилл сирот. Поведение не менялось.
- **14b-1 — спина мультитенанта** ✅ (коммит `3792930`): меню пускает любого
  зарегистрированного владельца и показывает только его каналы; онбординг и
  «➕ Добавить канал» проставляют владельца (ловушка 14a закрыта); экран
  «➕ Пригласить владельца» у супервладельца; уведомления онбординга — владельцу канала.
  После 14b-1 триал можно запускать: тестер ведёт свой канал напрямую, без одобрения.
- **14b-2 — одобрение по владельцу** ✅ (коммит `51ee05f`): ядро `core/approval/access.ts`
  (`resolveOwnerTarget` / `canActOnChannel`) + `getOwnerTelegramIdByChannelId`; превью
  одобрения и все служебные уведомления рантайма (нет постов, канал не задан, сбой
  AI-подхвата, разовый пост, модерация комментов, окончание контента, недельный отчёт) —
  владельцу КАНАЛА; approval-композер пускает владельцев, но каждый `ap:*` и `api`/`apre`/
  `apdel` проверяют принадлежность `PendingPost` каналу нажавшего; «📊 Отчёт» — по
  выбранному каналу. `postButtons` проверен: admin-фильтров нет (кнопки публичные).
  Исходный план подшага (выполнен):
  служебные уведомления владельцу канала. Конкретно:
  - превью одобрения (`PostingDeps.adminId` / `AiPostApprovalDeps.adminId` в
    `admin/index.ts`, `approvalService`, планировщике) → владельцу КАНАЛА, а не
    супервладельцу; approval-композер (`features/approval`) пускает владельцев
    (сейчас фильтр `ctx.from?.id === adminId`);
  - результат «🤖 AI-пост» (сообщение об ошибке генерации) — нажавшему;
  - служебные уведомления рантайма по каналу (окончание контента и т.п.) — владельцу канала;
  - `postButtons` — предсказания и так публичные, проверить, что admin-фильтров нет.
- **14b-bis** (отложен): свой bot-token на клиента.
- **14c–14e** (отложены): веб-кабинет, биллинг/касса, тарифы и поля триала на `Owner`.

## Безопасность (обязательный пункт)

Переключатель каналов берёт канал из нажатой кнопки — нужно проверять принадлежность
текущему владельцу, иначе крафтнутый callback выберет чужой. ✅ Закрыто в 14b-1
структурно: индекс в списке `listChannelsByOwner` + `pickSelectedId` отбрасывает id вне
списка. В 14b-2 такой же принцип применить к approval-callback'ам (`ap:*` несут id
`PendingPost` — проверять, что пост принадлежит каналу владельца).
