# Отчёт проверки Saenc BOT (MasterPost)

Дата: 2026-07-03. Проверка по «Личному контуру» (TrustLayer).

## Вердикт: ДА, С ОГОВОРКАМИ

Код заметно лучше типичного AI-кода: чистая архитектура (композеры, репозитории, сервисы), env через zod, 16 файлов тестов, секреты не в коде, фолбэки при отправке. На реальных пользователей выпускать можно **после исправления одного критичного пункта**.

---

## 🔴 Критично (1)

### 1. Нет глобального обработчика ошибок бота (`bot.catch`)

Бот работает на long polling (`bot.start()`), но нигде не вызван `bot.catch()`. Любая необработанная ошибка в хендлере (например, упала БД в момент нажатия кнопки подписчиком) отклонит промис `bot.start()`, а так как `main()` вызван через `void main()` — процесс умрёт от unhandled rejection.

**Простыми словами:** один сбой в неудачный момент — и бот молча лежит, пока вы не заметите.

**Исправление** (5 строк в `src/index.ts` перед `bot.start`):
```ts
bot.catch((err) => {
  logger.error({ err: err.error, update: err.ctx.update.update_id }, "ошибка обработки апдейта");
});
```
Плюс страховка процесса:
```ts
process.on("unhandledRejection", (err) => logger.error({ err }, "unhandled rejection"));
```

## 🟡 Важно (3)

### 2. Нет авто-повтора при лимитах Telegram (429)

Планировщик постит в несколько каналов + рассылает предсказания в личку. При росте нагрузки Telegram вернёт `429 retry_after`, и сообщение просто потеряется (залогируется как ошибка). Решается плагином без изменения кода:
```bash
npm i @grammyjs/auto-retry
```
```ts
import { autoRetry } from "@grammyjs/auto-retry";
bot.api.config.use(autoRetry());
```

### 3. Уязвимости в dev-зависимостях (1 critical, 1 high)

`npm audit`: критическая в vitest (`@vitest/mocker`) и high в vite. Обе — **только dev-инструменты**, в работающий бот не попадают, но лучше закрыть: `npm i -D vitest@4` (мажорное обновление, прогнать тесты после).

### 4. Состояние меню админа — в памяти

`channelContext.ts`: `const selected = new Map<number, string>()` — выбранный канал теряется при перезапуске. Затрагивает только вас (админа), не подписчиков: после рестарта заново выбрать канал в меню. Осознанный компромисс — ок, просто знать.

## 🟢 Мелочь (2)

- `parse_mode: "Markdown"` (легаси): заголовок поста с символом `*` или `_` ломает разметку. Фолбэк «отправить без разметки» есть — пост не пропадёт, но выйдет без жирного. Надёжнее MarkdownV2 с экранированием.
- Тесты в моей среде не запустились из-за node_modules, собранных под Windows (rollup native). На вашей машине прогоните `npm test` — должны быть зелёные.

## Что уже хорошо (не трогать)

- Ошибка одного канала не роняет автопостинг остальных (try/catch на канал)
- Битое фото/кривой Markdown не «съедают» пост — публикация текстом
- Pexels с таймаутом 8с и без исключений наружу
- Кулдауны и pending-посты в БД, переживают перезапуск
- `.env` в `.gitignore`, секретов в коде нет
- Health-сервер для хостинга, догон пропущенных слотов после простоя

## Порядок действий

1. `bot.catch` + `unhandledRejection` (10 минут) — до выпуска на людей
2. `auto-retry` плагин (10 минут)
3. На хостинге включить авто-рестарт процесса (docker `restart: always` / systemd)
4. `npm i -D vitest@4`, прогнать тесты — в любой удобный момент
